## Custom Project

### Dataset

For my custom project, I used the sales dataset in `data/sales.csv`. The dataset contains sales order records that are streamed through Kafka by the producer.

Each sales record includes fields such as `order_id`, `datetime`, `region_id`, `currency_code`, `product_id`, `unit_price`, `quantity`, `is_online`, `customer_id`, `device_type`, `payment_method`, `referral_source`, `discount_code`, and `customer_note`.

The producer streams from `sales.csv`. The other CSV files in the `data/` folder are static reference tables. These include `regions.csv`, `products.csv`, `currencies.csv`, and `discount_codes.csv`. The consumer uses the region reference table to calculate tax and enrich the streamed records.

I used the original sales dataset and customized the consumer processing logic to simulate an omnichannel retail fulfillment routing workflow.

### Kafka Messages

The Kafka producer reads records from `data/sales.csv` and sends each sales record as a message to the Kafka topic.

The Kafka topic used in my run was `product-sales-case`. Each Kafka message contains one sales order record from the CSV file. The producer used the region value as the message key, such as `US-TX`, `US-CA`, `US-MO`, `CA-ON`, or `CA-QC`.

I did not change the producer message structure. Instead, I customized the consumer so the consumed sales messages could be enriched with omnichannel fulfillment fields after they were received from Kafka.

### Consumer Processing

I created a custom consumer named `kafka_consumer_kiruthikaa.py`.

The consumer receives sales messages from Kafka, validates required fields, computes derived sales values, updates the live chart, writes processed records to CSV, and stores valid records in DuckDB.

My custom consumer extends the original processing by adding omnichannel retail fulfillment routing fields. These fields include `customer_name`, `customer_email`, `order_type`, `fulfillment_source_type`, `fulfillment_location`, `total_sales_amount`, `return_eligible`, and `return_status`.

The consumer uses existing message values such as `region_id`, `quantity`, `is_online`, `device_type`, and `total` to classify orders. For example, an online order with a quantity of 3 is classified as an `ENDLESS_AISLE_ORDER` and routed to a warehouse location such as `LVL`. Mobile or tablet orders with quantity 1 are classified as `BOPIS_ORDER` and routed to a store pickup location such as `LV`, `SF`, `LA`, or `BUR`.

The processed output is written to `data/output/omnichannel_fulfillment_kiruthikaa.csv`. The chart output is written to `data/output/omnichannel_sales_chart_kiruthikaa.png`.

### Experiments

For Phase 4, I copied the original consumer into a new file named `kafka_consumer_kiruthikaa.py`. I made an initial technical modification by changing the output CSV and chart names so my custom output files were separate from the original example outputs.

For Phase 5, I applied the Kafka streaming skills to a new supply chain and retail problem. I added omnichannel fulfillment routing logic to classify streamed sales records as `BOPIS_ORDER`, `ENDLESS_AISLE_ORDER`, `SHIP_TO_CUSTOMER`, or `POS_STORE_ORDER`.

I also added fulfillment source and location logic. Store pickup orders are routed to store locations such as `LA`, `BUR`, `LV`, or `SF`. Endless Aisle and ship-to-customer orders are routed to warehouse locations such as `LVL` or `ONT`.

### Results

When I ran the producer, it sent sales messages to the Kafka topic successfully. When I ran my custom consumer, it consumed the messages, calculated subtotal, tax amount, and total, and added the custom omnichannel routing fields.

The terminal output showed the custom classifications, including examples such as `ENDLESS_AISLE_ORDER` with `WAREHOUSE` fulfillment and `BOPIS_ORDER` with `STORE_PICKUP` fulfillment.

The output CSV file included the original sales fields, calculated tax and total fields, and my custom omnichannel fulfillment fields. This confirmed that the consumer modification worked and that the Kafka messages were successfully processed into business-focused output.

### Interpretation

This Kafka streaming workflow showed how sales records can move from a CSV file into a Kafka topic and then into a Python consumer for processing.

The original example demonstrated a complete streaming pipeline with validation, derived fields, visualization, CSV output, and DuckDB storage. My custom work extended that pipeline by adding omnichannel retail fulfillment intelligence.

The main change from the original example was that my consumer did not only calculate sales totals. It also classified each order into an operational fulfillment scenario and identified whether the order should be fulfilled from a store or a warehouse.

This helped me understand how streaming analytics can support real-time business decision-making. In a retail or supply chain environment, a similar workflow could help operations teams identify BOPIS orders, Endless Aisle orders, warehouse fulfillment orders, and return eligibility as messages arrive.

The business intelligence gained from the consumed messages includes order type, fulfillment source, fulfillment location, tax amount, total sales amount, and return status. This turns raw sales messages into more useful operational data for omnichannel fulfillment planning.
